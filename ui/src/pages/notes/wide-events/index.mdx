I like to think about how to get operational insights into services I host. It's become kind of a
hobby to be honest. Maybe this whole thing is just bike shedding. What better way to avoid the
crushing reality that an actual project you had in mind doesn't live up to your expectations than by
not even attempting it and instead building infrastructure around your existing applications. Maybe
that's a topic for therapy.

Anyhow, I realized during my latest manic observability episode that I barely even remember how I
got here. So for posterity, I'm outlining the history and motivations for all things metrics in this
note.

## Act 1: Metrics and Prometheus

A couple of years ago I got the itch to play around with embedded devices. I got myself an ESP32 and
a CO2 sensor and got to work. With a little ellbow grease, I had the current CO2 concentration
displayed on a small seven segment display. Quickly, I graduated to wanting to store and chart this
data.

At this point I had little knowledge on how to approach this, so I went the well-trodden path.
Prometheus to store time series data and Grafana to visualize it. Since my microcontroller was not
internet accessible, I had to put a [push gateway](https://github.com/prometheus/pushgateway) in
front of Prometheus to allow pushing data to the system, instead of having Prometheus scrape the
metrics off the device regularly.

Around this time I also built and deployed this website. Wanted to collect visitor statistics
[here](/notes/performance/) thoughprocess to store it separately is what spawned the next iteration.

## Act 2: Metrics with DuckDB

-> observatory

At some point I got fed up by the limitations of Prometheus and Grafana and also the need to be to
understand PromQL. I had these moments where I wanted to make a moderately sophisticated chart,
let's say grouping, let's say averaging data into time buckets and then showing them, which I knew
how to do in just about every language under the sun, but I just could not bring myself to learn
PromQL, such a domain specific language for such a small problem.

It was also an interesting engineering question how one could rebuild such a stack like Prometheus
and Grafona. Namely, how do you store data efficiently for such a purpose, and then how do you query
it efficiently. Further, how would you allow an end user to specify queries in the front end and
have them securely and efficiently executed on the back end.
[unstructured data](/notes/unstructured-data/)

It was for this purpose that I started looking into WebAssembly as an isolation mechanism to run
untrusted code from the user on the backend. [wasm](/notes/wasm-benchmark/)

[duckdb](/notes/sqlite-vs-duckdb/) I quickly landed on DocDB as a storage engine as that proved to
be significantly faster than SQLite, and I was really in a downsizing mood here as I tried to get
away from the client server approach of Prometheus and Grafona. So ultimately I ended up building my
own custom stack here that was accepting arbitrary JSON input from systems, and I was also building
a custom frontend in Solid.js to show this data. The approach of using WebAssembly that the user
could run inside the browser, or the user could write inside the browser and would get compiled and
run on the server ultimately never took flight. But it was an interesting excursion, and I learned a
bunch doing it.

At this point, I was still only using this application for storing metrics as well as GPS location
data. I was still missing a comprehensive solution for access logs from my website. And in the
process of reading into this and thinking about what kind of solution I could come up with here, I
stumbled across tracing. Now at this point it was my belief that tracing could replace all three
traditional pillars of observability, in particular logs are simply events emitted within some span,
and metrics can be generated from attributes encompassed in tracing events. So at this point, I
started looking into building my own open telemetry collector such that I could hook into the
existing ecosystem around tracing. This proved to be rather difficult as the documentation around
open telemetry is severely lacking in my opinion. This line of thinking ultimately led me down
towards the next iteration of observability.

## Act 3: Tracing with ClickHouse

-> events

Since I started hosting my own side projects and making them available to the internet, I've craved
introspection into those services. Not logs for debugging errors, I rarely get those (my Rust
evangelism compels me to mention that a strict language really helps in cutting down runtime
errors). Instead I mean operational indicators: How many visitors do I get on my website, which
notes are most accessed? Which endpoints have high latency, does a particular user cause a
disproportionate load? And most importantly, which Wordpress vulnerability is all the rage with the
crawlers right now?

After falling in love with the idea of having a consolidated system that simply treats traces and
nothing else. I started looking around for the best technologies to handle this. And Clickhos showed
up as a series contenter. So this is what I ended up doing. I instrument my service that is hosting
this website using the Rust tracing crate. Then I set up an open telemetry collector for Clickhalls.
And then I had my Rust service simply dump all of this tracing data to this collector, and the
collector in turn writes to Clickhauz. At no point do I need to touch any database SDK to directly
write to the database. Even the endpoints I expose out of an additional Rust service that I use to
ingest the IoT and the GPS data, this service does not actually write directly to Clickholds. It
simply uses the tracing crate to generate spans that go via the collector into the database.
[clickhouse](/notes/clickhouse/) [tracing annoyance](/notes/otel/)

If this sounds hacky, that's because it is. Even small sensor readings that could be stored in,
let's say, eight bytes or 16 bytes, blow up to a full database row of a rather complicated database
schema needed for open telemetry tracing. But the simplicity of it is undeniable.

## Act 4: Wide events with Parquet

-> events v2

[blog](https://jeremymorrell.dev/blog/a-practitioners-guide-to-wide-events/)

So I have been doing some reading again. And this time I've stumbled across the next interesting
concept. Just like I did with tracing in the last iteration, this time it is wide events. The idea
is very simple. As a backend system is working towards responding to a request, you build up
properties in a context that you can ultimately at the time you respond to the request collect into
a wide event and send towards the observability system. One thing to note is that this is
significantly simpler than tracing. There are no sophisticated protocols, nothing. It's a simple
chase and blob. The second thing to note is that you can still use this to generate any metrics that
you would like. You can count requests per second, you can aggregate errors per user agent, whatever
you want. You have all the data you could possibly want in these events.

If the query engine is performant enough to pull out arbitrary fields from these JSON objects, we
can put whatever data we want into that system and get nice dashboards. For instance, the IoT data
that I want to ingest will also be simple JSON blocks that I store. The GPS data from my phone, the
same.

If you haven't already, you should check out the last note where you can read a bunch of my thought
process towards this new system. To summarize, the idea is to radically simplify from an active
database down to simple parquet files. At this point, I'm planning to use DocDB as the writer as
well as the query engine to read and write those files. But ultimately, the docdb could easily be
switched out for a different query engine at any point. The simplicity of the system is that there
would be a single Rost service that accepts arbitrary JSOS and writes this directly to a parquet
file. [parquet](/notes/duckdb-parquet/)
